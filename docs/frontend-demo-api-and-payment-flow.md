# Frontend Demo API and Payment Flow

This document defines the final API surface needed for a simple frontend demo of the e-commerce saga flow.

The goal is not to build a full production storefront. The goal is to provide a small frontend that demonstrates the distributed order-processing saga clearly:

- Product catalog
- Customer selection/creation
- Order creation
- Product reservation
- Payment confirmation/failure
- Order status transitions
- Notification through Maildev
- Observability links such as Zipkin, Eureka, Keycloak, database UIs, etc.

---

## Customer Service

No changes are required.

```http
GET    /api/v1/customers
POST   /api/v1/customers
GET    /api/v1/customers/{id}
PUT    /api/v1/customers/{id}
DELETE /api/v1/customers/{id}
GET    /api/v1/customers/exists/{id}
```

---

## Product Service

Final frontend-facing endpoints:

```http
GET    /api/v1/products
POST   /api/v1/products
GET    /api/v1/products/{id}
PUT    /api/v1/products/{id}
POST   /api/v1/products/{id}/stock-adjustments
```

The stock adjustment endpoint should represent a stock delta, not an absolute stock replacement.

Remove this endpoint from the public API and from the service, it should not exist:

```http
POST /api/v1/products/purchase
```

Product purchase/reservation should be driven by the saga, not by a direct public purchase endpoint.

---

## Order Service

Final frontend-facing endpoints:

```http
GET  /api/v1/orders
POST /api/v1/orders
GET  /api/v1/orders/{id}
GET  /api/v1/orders/{id}/order-lines
```

Remove or stop exposing duplicate/non-versioned endpoints such as:

```http
GET /{id}
```

The order detail endpoint returns the order header/state. Order lines remain exposed through the dedicated order-lines endpoint.

Recommended `OrderStatus`:

```java
public enum OrderStatus {
    PRODUCT_RESERVATION_PENDING,
    AWAITING_PAYMENT,
    CONFIRMED,
    PRODUCT_RESERVATION_FAILED,
    PAYMENT_FAILED
}
```

`AWAITING_PAYMENT` replaces the previous `PRODUCT_RESERVED` status. It expresses the business state more clearly: stock has already been reserved and the order is waiting for the payment result.

---

## Payment Service

Payment Service should be exposed for demo payment simulation and payment state inspection.

Final frontend-facing endpoints:

```http
GET  /api/v1/payments
GET  /api/v1/payments/{id}
GET  /api/v1/payments/orders/{orderId}
POST /api/v1/payments/demo/orders/{orderId}/confirm
POST /api/v1/payments/demo/orders/{orderId}/fail
```

Do not expose direct payment creation as a normal public API:

```http
POST /api/v1/payments
```

Payment creation should remain saga-driven through Kafka.

Recommended `PaymentStatus`:

```java
public enum PaymentStatus {
    PENDING,
    CONFIRMED,
    FAILED
}
```

The `/demo` endpoints simulate the external payment provider result. They are not intended to represent a real payment provider integration.

---

## Notification Service

No frontend-facing endpoints are required.

Notification remains fully event-driven.

The frontend should expose a visible link to Maildev so the user can inspect the emails generated by the saga.

## New Payment Flow

```text
1. Frontend creates an order through Order Service.

2. Order Service creates the order with status PRODUCT_RESERVATION_PENDING.

3. Order Service publishes OrderCreated.

4. Product Service consumes OrderCreated.

5. Product Service tries to reserve stock.

6. If stock reservation fails:
   - Product Service publishes ProductReservationFailed.
   - Order Service consumes it.
   - Order status becomes PRODUCT_RESERVATION_FAILED.
   - Notification flow can continue as currently designed.

7. If stock reservation succeeds:
   - Product Service publishes ProductReservationSucceeded.
   - Order Service consumes it.
   - Order status becomes AWAITING_PAYMENT.
   - Order Service publishes PaymentRequested.

8. Payment Service consumes PaymentRequested.

9. Payment Service creates a Payment with status PENDING.
   It must not auto-confirm the payment in demo mode.

10. Frontend polls:
    - GET /api/v1/orders/{id}
    - GET /api/v1/orders/{id}/order-lines
    - GET /api/v1/payments/orders/{orderId}

11. When Order status is AWAITING_PAYMENT and Payment status is PENDING, the frontend shows:
    - Confirm payment
    - Fail payment

12. If the user confirms payment:
    - Frontend calls POST /api/v1/payments/demo/orders/{orderId}/confirm.
    - Payment Service updates payment status to CONFIRMED.
    - Payment Service publishes PaymentConfirmed.
    - Order Service consumes PaymentConfirmed.
    - Order status becomes CONFIRMED.
    - Notification flow continues as currently designed.

13. If the user fails payment:
    - Frontend calls POST /api/v1/payments/demo/orders/{orderId}/fail.
    - Payment Service updates payment status to FAILED.
    - Payment Service publishes PaymentFailed.
    - Order Service consumes PaymentFailed.
    - Order status becomes PAYMENT_FAILED.
    - Product compensation/stock release should continue if already implemented.
    - Notification flow continues as currently designed.
```

## Frontend Scope

The frontend should remain intentionally simple.

It should display:

```text
Products
Customers
Orders
Order detail
Order lines
Order status
Payment status
Payment confirm/fail actions
Static dev links: Maildev, Zipkin, Eureka, Keycloak, database UIs, etc.
```

The most important frontend page is the order detail page.

It should show:

```text
Order ID
Customer information or customer ID
Order status
Order lines
Payment status
Confirm payment / Fail payment actions when applicable
A link to Maildev to inspect notification emails
A link to Zipkin to inspect distributed traces
```

---

## Recommended Frontend Flow

```text
1. User opens the product catalog.
2. User selects products and quantities.
3. User selects or creates a customer.
4. User creates an order.
5. Frontend redirects to the order detail page.
6. Frontend polls order and payment state.
7. If the order reaches AWAITING_PAYMENT, frontend shows payment action buttons.
8. User confirms or fails the payment.
9. Frontend keeps polling until the order reaches a final state:
   - CONFIRMED
   - PAYMENT_FAILED
   - PRODUCT_RESERVATION_FAILED
10. User can inspect Maildev and Zipkin through always-visible dev links.
```

---

## Final Design Decision

Payment should be exposed only for demo-oriented payment result simulation and state inspection.

Order Service should not call Payment Service through OpenFeign just to confirm or fail a payment. That would add unnecessary synchronous coupling and make the choreographed saga harder to reason about.

The cleaner demo flow is:

```text
Create order:
Frontend -> Order Service -> Kafka -> Product Service -> Kafka -> Order Service -> Kafka -> Payment Service

Confirm/fail payment:
Frontend -> Payment demo endpoint -> Kafka -> Order Service -> Kafka -> Notification Service
```

This keeps Payment responsible for payment results while still allowing the frontend to demonstrate both successful and failed saga paths.
